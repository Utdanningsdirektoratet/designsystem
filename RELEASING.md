# Releasing

This document describes how the automated release pipeline works and how to
bootstrap the first stable release.

For day-to-day workflow (which branch to target with your PR), see the
[README](README.md#hvordan-publisere-en-ny-versjon).

## How it works

Releases are fully automated via
[semantic-release](https://semantic-release.org/). When commits are pushed
(or merged) to a configured release branch, the CI workflow:

1. **Determines the version bump** from conventional commit messages
   (`feat` → minor, `fix`/`perf`/`revert` → patch, `BREAKING CHANGE` → major).
2. **Generates release notes** including commit body text and breaking change
   explanations.
3. **Syncs the version** into every publishable `package.json`
   (`@udir-design/{react,css,icons,symbols,theme}`).
4. **Stages** all packages on the npm registry (awaiting approval).
5. A human **approves** the staged packages, promoting them to the public registry.
6. **Creates a GitHub release** with the generated notes.
7. **Tags** the commit as `v<version>`.

Pull requests to release branches get a **changelog preview** posted as a PR
comment, so reviewers can see exactly what will be released.

### Branch → channel mapping

| Branch pattern                  | npm tag  | Version format | Purpose                             |
| ------------------------------- | -------- | -------------- | ----------------------------------- |
| `release/latest`                | `latest` | `1.2.3`        | Stable releases                     |
| `release/beta`                  | `beta`   | `1.2.3-beta.N` | Beta pre-releases (due for removal) |
| `release/N.x` / `release/N.N.x` | `N.x`    | `1.13.2`       | Maintenance releases                |

### What triggers a release

Only these commit types cause a version bump:

- **`feat:`** — minor version bump
- **`fix:`** — patch version bump
- **`perf:`** — patch version bump
- **`revert:`** — patch version bump (when the body contains
  `"This reverts commit <hash>"`, as generated by `git revert`)
- **`BREAKING CHANGE` footer or `!` indicator** — major version bump

Other types (`chore`, `ci`, `docs`, `refactor`, `test`, `build`, `style`) do
**not** trigger a release. They are included in the commit history but omitted
from the changelog.

> **Note on reverts:** The commit-analyzer matches reverts via the
> `commit.revert` metadata (set by the parser when `"This reverts commit <hash>"`
> appears in the body), not by `type: 'revert'`. Always use `git revert` rather
> than writing `revert:` subjects manually. If reverting a feature is a breaking
> change, add `!` or a `BREAKING CHANGE` footer to trigger a major bump.

### Staged publishing

Packages are published using [`pnpm stage publish`](https://pnpm.io/cli/stage),
which uploads them to a staging area on the npm registry without making them
publicly available. A human must then approve each staged version:

```bash
# List staged packages
pnpm stage list

# Inspect a specific staged version
pnpm stage view <stage-id>

# Approve (requires 2FA / OTP)
pnpm stage approve <stage-id> --otp <otp>

# Or reject
pnpm stage reject <stage-id>
```

This adds a manual gate between CI automation and public availability.

### Changelog format

Each changelog entry includes:

- The commit subject (first line)
- The commit scope (if any), bolded
- A link to the commit on GitHub
- The commit body text (indented), with footer tokens stripped
- A dedicated **⚠ BREAKING CHANGES** section at the top when applicable

This matches the format produced by the former Nx-based `changelog-renderer.ts`.

## Bootstrapping (one-time procedures)

### 1. Bootstrap git notes for existing tags ✅ Done

semantic-release tracks which npm channel a tag was published on using git
notes (`refs/notes/semantic-release-<tag>`). Tags created by the old Nx-based
release system didn't have these notes. A one-time bootstrap script was run to
derive the channel from each tag's semver prerelease identifier and attach the
note to the underlying commit. The notes have been pushed to the remote.

For new tags, semantic-release creates and pushes notes automatically.

### 2. Bootstrap v1.0.0 (stable baseline)

`release/latest` is already set to the same commit as `v1.0.0-beta.33`. The
first merge into `release/latest` will trigger semantic-release, which will
create `v1.0.0` with tag, git note, and GitHub release automatically.

The generated changelog for this first release will include the entire commit
history (since there is no prior stable tag). Edit the GitHub release
description afterwards to something sensible, e.g. _"First stable release.
See beta releases for earlier changelog."_

After this, semantic-release handles everything. The next commit merged into
`release/latest` (e.g. `feat: add dialog`) will produce `v1.1.0` with a clean,
incremental changelog.

## Local testing

To test locally, create a sandbox bare repo with a tagged baseline that
semantic-release can recognise. Three things are needed:

1. A **tag** at the baseline commit (e.g. `v1.0.0`).
2. A **git note** on that commit so semantic-release knows it was a release
   (`refs/notes/semantic-release-v1.0.0` → `{"channels":[null]}`).
3. The **local branch** `release/latest` must point at (or ahead of) the tag,
   because `git tag --merged release/latest` uses the local ref.

```bash
# 1. Create a temporary bare repo
git init --bare /tmp/test-remote.git

# 2. Push current HEAD as the release branch, with all tags
git push /tmp/test-remote.git HEAD:refs/heads/release/latest --tags
git --git-dir=/tmp/test-remote.git symbolic-ref HEAD refs/heads/release/latest

# 3. Tag HEAD as the baseline and add a semantic-release git note
git tag v1.0.0 HEAD
git notes --ref semantic-release-v1.0.0 add -f -m '{"channels":[null]}' HEAD
git push /tmp/test-remote.git v1.0.0 "refs/notes/semantic-release-v1.0.0"

# 4. Sync the local release/latest branch to HEAD
git branch -f release/latest HEAD

# 5. Create test commits, then push and run
git commit --allow-empty -m "feat: test feature"
git branch -f release/latest HEAD
git push /tmp/test-remote.git HEAD:refs/heads/release/latest --force

./@internal/ci/bin/semantic-release.ts \
  --branch release/latest \
  --preview-changelog \
  --repository-url file:///tmp/test-remote.git

cat CHANGELOG.md

# Cleanup
git reset HEAD~1
git tag -d v1.0.0
git notes --ref semantic-release-v1.0.0 remove HEAD 2>/dev/null
git branch -f release/latest origin/release/latest
rm -rf /tmp/test-remote.git CHANGELOG.md
```

> **Why the local branch must be synced:** semantic-release resolves prior
> release tags with `git tag --merged release/latest`, which uses the
> **local** branch ref — not the remote. If the local `release/latest`
> points to an older commit, the baseline tag won't be found.

## Troubleshooting

### Cleaning up after a failed release

If the CI publish step fails (e.g., npm auth error, network issue), semantic-release
may have already pushed a git tag and git note before the failure. These must be
removed before re-running, otherwise semantic-release considers the version already
released.

1. **Identify the version** from the failed run's logs (e.g., `v1.0.0-beta.34`).

2. **Check what exists on the remote:**

   ```bash
   git ls-remote origin refs/tags/v1.0.0-beta.34
   git ls-remote origin 'refs/notes/semantic-release-v1.0.0-beta.34'
   ```

3. **Delete the tag and note from the remote:**

   ```bash
   git push origin :refs/tags/v1.0.0-beta.34 :refs/notes/semantic-release-v1.0.0-beta.34
   ```

4. **Delete the local tag (if fetched):**

   ```bash
   git tag -d v1.0.0-beta.34 2>/dev/null
   ```

After cleanup, re-run the "Publish semantic release" job in the GitHub Actions run that failed.
semantic-release will re-attempt the same version.

### OIDC "permission denied" errors

If you see `OIDC permission denied for this action` when publishing, check the
trusted publisher configuration on npmjs.org for each `@udir-design/*` package:

- **Allowed actions** must include `npm stage publish` (not just `npm publish`).
- **Workflow filename** must be `ci.yml` (the top-level calling workflow), not
  `release.yml`, because npm validates the caller in `workflow_call` chains.
